# Adversarial Review

## いつ使うか

- 破壊的変更、広い影響範囲、複雑な境界条件、運用回帰の懸念があり、親が高リスク変更と判断したとき
- review 対応や追加コミット前に、独立した追加レビューが欲しいとき
- 説明やテスト結果だけでは確信しきれないとき

## いつ使わないか

- まだ差分が固まっていないとき
- 呼び出し元がレビュー結果を評価したり実装へ戻したりできないとき
- 実装や修正までこの skill 単体にやらせたいとき
- 軽微な変更、局所変更、既存パターンへの単純追従で、self-review と必要な検証で十分なとき

## 責務

- 差分を読み、findings / open question / 残リスクを返す
- 対応判断、実装修正、再検証、コミットは行わない

## 正本

- repo 共通ルールは `AGENTS.md` を正本とする

## 判断ルール

- 呼び出し元は、高リスク変更かどうかを先に判断し、必要時だけこの skill を呼ぶ
- 懸念は重大度順に整理し、差分や実行結果に根拠があるものを優先する
- 仕様未確認や検証不足は、その不足自体をリスクとして扱う
- 「たぶん大丈夫」で閉じず、どの条件なら危ないかを具体化する
- 根拠が弱い懸念は、断定せず open question として分ける
- サブエージェントが使える環境では、呼び出し元と独立した文脈のサブエージェントにレビューだけを依頼してよい
- サブエージェントにはコード修正、コミット、push をさせず、findings の報告に専念させる

## 手順

1. 現在 branch とレビュー対象の差分範囲を確認する
2. サブエージェントが使える環境なら、レビュー専用の独立サブエージェントに差分確認を依頼する
3. `main` または作業元 branch、必要なら直近コミットとの差分を読む
4. 変更ファイルごとに、主題から外れた変更や説明不足を洗う
5. 挙動破壊、境界条件、データ整合性、運用回帰の順で懸念を探す
6. 検証コマンドと実行結果を確認し、未検証箇所を洗い出す
7. 懸念を `要対応 / open question / 残リスク` に分けて整理する
8. 呼び出し元が判断しやすいよう、各懸念に根拠と影響を書く

## 停止条件

- 差分の主題が複数に割れていて、レビュー観点を定められない
- 差分や検証結果が不足し、根拠を持って懸念を述べられない
- 修正指示や実装まで求められており、レビュー専用の責務を保てない

## 出力

- 重大度順の findings
- 各 finding の根拠
- open question
- 残リスク
- 呼び出し元側で判断・実装すべき事項
