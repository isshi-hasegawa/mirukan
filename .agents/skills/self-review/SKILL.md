# Self Review

## 目的

完了報告や PR 作成の前に、差分と検証結果を見直して重大な懸念を先に潰し、未確認事項を明示できる状態へ整える。

## 対象

この skill は次の作業を扱う。

- 作業ブランチと base branch の確認
- 差分の棚卸し
- 変更内容のセルフレビュー
- 検証コマンドと結果の整理
- 未実施の確認と残リスクの明示
- 重大な懸念が見つかった場合の差し戻し判断

## 正本

- リポジトリ固有の運用ルールは `AGENTS.md` を正本とする
- 実装後チェックと検証方針は `AGENTS.md` に従う

## 利用ツール

- 差分、コミット、状態確認には `git` コマンドを使う
- 検証には `vp` コマンドを使う
- `pnpm` / `npm` は使わない

## 前提ルール

- セルフレビューは完了報告の前に実施する
- `main` ではなく作業ブランチ上で行う
- レビュー対象は未コミット変更を含む作業中の差分全体とする
- 変更内容の説明は実際の差分と検証結果に基づける
- 未確認事項がある場合は隠さず明示する
- 重大な懸念が見つかった場合は、修正してからコミットまたは追加コミットする

## 実行前チェック

1. `git branch --show-current` で現在 branch を確認する
2. `git status --short --branch` で作業状態を確認する
3. base branch を確認する。原則は `main` とする
4. `git diff --stat <base>...HEAD` と必要なら `git diff <base>...HEAD` を確認する
5. 未コミット変更がある場合は、それもレビュー対象に含める
6. 実行済みの検証コマンドと未実施の検証を整理する

## 判断ルール

- 差分に不要な変更が混ざっている場合は、先に整理する
- 回帰リスクは正常系より先に、破壊しうる箇所を優先して確認する
- レビュー観点は見た目より、挙動の破壊、運用上の影響、境界条件、説明不足を優先する
- テスト追加や更新が必要なのに不足している場合は懸念として扱う
- 実行した検証だけを「確認済み」として扱う
- 未実施の確認は、問題なしと扱わず残リスクとして明示する

## 推奨手順

1. 現在 branch とワークツリー状態を確認する
2. `main` または作業元 branch に対する差分を確認する
3. 変更ファイルごとに、不要変更が混ざっていないかを見る
4. 実装の主目的に対して、挙動の破壊や回帰がないかを確認する
5. 影響範囲の周辺で、境界条件や運用上の懸念がないかを確認する
6. テスト、ビルド、lint、knip など必要な検証の実施有無を整理する
7. 未実施の検証と、その理由を整理する
8. 重大な懸念があれば修正に戻る
9. 懸念がなければ、確認した内容と残リスクを報告にまとめる

## 差分確認の観点

- 変更ファイルは今回の主題に沿っているか
- 生成物や incidental な変更が紛れ込んでいないか
- 命名、コメント、説明が差分の実体と合っているか
- 削除やリファクタリングで副作用が出ていないか
- 設定変更や依存変更で、周辺に影響が波及していないか

## レビュー観点

- 正常系だけでなく失敗系や境界条件も破綻しないか
- 既存の呼び出し元、利用者、運用フローに破壊的変更が入っていないか
- テストが仕様変更に追従しているか、または不足していないか
- ドキュメント変更なら、運用ルールや手順と矛盾していないか
- 変更内容の説明が足りず、後続作業者が誤読しないか

## 検証整理の観点

- 実行したコマンドを列挙できるか
- コマンド結果が成功か失敗かを説明できるか
- 実行していない検証がある場合、その理由を説明できるか
- 依存追加、削除、リファクタリング、ファイル削除では `vp run knip` の要否を検討したか
- テストがある変更では `vp test` の対象範囲が妥当か

## 停止条件

次の場合は自動で完了扱いにせず停止する。

- 差分の主題が複数に割れていて妥当性を判断しにくい
- 不要な変更が混ざっている
- 重大な挙動回帰の可能性を否定できない
- 必要な検証が未実施で、残リスクとして片付けるには重すぎる
- レビュー結果の説明根拠が差分や実行結果から取れない

## 出力

レビュー後は次を報告する。

- 懸念の有無
- 見つかった問題があれば、その内容
- 実行した検証コマンドと結果
- 未実施の検証
- 残リスク
