# Self Review

## いつ使うか

- 実装が完了し、コミットや PR 作成の前に差分を点検するとき
- [pr-followup](../pr-followup/SKILL.md) の修正後に、コミット前の最終確認をするとき
- 追加の見直しが必要で、自分の手元で差分と検証結果を再点検するとき

## いつ使わないか

- まだ実装途中で差分が確定していないとき
- 差分がないとき
- 独立した批判的レビューだけを追加したいとき

## 責務

- 差分と検証結果を見直し、妥当な懸念があればそのまま修正と再確認まで進める
- 実装完了後の必須 review として、コミットや PR 作成に進める状態を作る

## 正本

- repo 共通ルールは `AGENTS.md` を正本とする

## 判断ルール

- 差分に不要な変更が混ざっている場合は、先に整理する
- 回帰リスクは正常系より先に、破壊しうる箇所を優先して確認する
- テスト追加や更新が必要なのに不足している場合は懸念として扱う
- 実行した検証だけを「確認済み」として扱う
- 未実施の確認は、問題なしと扱わず残リスクとして明示する
- 独立した別視点のレビューが欲しいだけなら [`adversarial-review`](../adversarial-review/SKILL.md) を使う
- 懸念を見つけたら、まず差分・仕様・実行結果から妥当性を判断する
- 妥当な懸念で、その場で安全に直せるなら修正・再検証・再レビューまで続ける
- 妥当性が低い、または前提不足で判断しきれない懸念だけを open question / residual risk として残す

## 手順

1. 現在 branch とワークツリー状態を確認する
2. 今回の self-review で点検する差分の範囲を決める
3. `main` または作業元 branch、必要なら直近コミットとの差分を確認する
4. 変更ファイルごとに、不要変更が混ざっていないかを見る
5. 実装の主目的に対して、挙動の破壊や回帰がないかを確認する
6. 影響範囲の周辺で、境界条件や運用上の懸念がないかを確認する
7. `AGENTS.md` に照らして必要な検証の実施有無を整理する
8. 未実施の検証と、その理由を整理する
9. 懸念があれば、差分・仕様・実行結果から妥当性を判断する
10. 妥当な懸念があり、その場で対応可能なら修正して必要な検証をやり直し、必要な範囲だけ再度 self-review する
11. 懸念が解消したら、確認した内容と残リスクを報告にまとめる
12. 判断保留の懸念だけが残る場合は、その根拠と不足情報を明記して止める
13. self-review が通過したら、`gh pr list --head <branch>` で対応 PR が未作成か確認し、未作成なら `pr-open` を呼ぶ。`pr-followup` 経由で呼ばれた場合は既に PR が存在するためスキップする

## 停止条件

- 差分の主題が複数に割れていて妥当性を判断しにくい
- 不要な変更が混ざっている
- 重大な挙動回帰の可能性を否定できない
- 必要な検証が未実施で、残リスクとして片付けるには重すぎる
- レビュー結果の説明根拠が差分や実行結果から取れない

## 出力

- 懸念の有無
- 見つかった問題があれば、その内容
- 妥当と判断して対応した問題があれば、その修正内容と再確認結果
- 実行した検証コマンドと結果
- 未実施の検証
- 残リスク
