# Feature TODO

未着手の機能追加、UX 改善、ユーザー価値ベースの改善候補を置く。
実装中のタスク管理ではなく、ユーザー向け価値を増やす候補の整理に絞る。

## 視聴モード分類ロジックの見直し

- 背景
  - `ガッツリ` と `じっくり` を中心に、現在の分類基準が体感とずれるケースがある
  - 再設計した四択 UI に対して、裏側の判定ロジックが追いついていないと選択体験の納得感が弱い
- 進め方の軸
  - 再生時間だけでなく、シリーズ/映画差や完走負荷など、既存スコアの使い方を再点検する
  - 各モードに入る代表例・外れる代表例を洗い出し、期待値ベースで閾値を調整する
  - UI 文言とロジック上の意味がずれないように、モード定義と表示文言をセットで見直す

## TMDb 活用機能の拡張

- 背景
  - 現状でも TMDb 検索、類似作品、おすすめ、シーズン取得、作品詳細は使えている
  - 一方で、取得したメタデータを「追加時」以外の体験へ広げる余地が大きい
  - 特に配信状況、公開日、シリーズ継続視聴、クレジット起点の探索は backlog 体験と相性がよい
- 進め方の軸
  - 追加モーダル内だけで完結させず、ボード上の意思決定に効く導線へ広げる
  - TMDb 由来データの鮮度管理が必要な機能は、再取得タイミングとキャッシュ方針もセットで決める
  - 似た価値の機能を同時に増やさず、まずは 1 画面で価値が伝わる候補から着手する
- 検討メモ
  - [TMDb Feature Ideas](../ideas/tmdb-feature-ideas.md)

## アカウント設定セルフサービスの整備

- 背景
  - 現状の認証導線はログイン、新規登録、パスワードリセットまでは用意できている
  - 一方で、ログイン後に登録メールアドレスを変更する導線は未実装
  - Supabase 側では `Change Email Address` テンプレートを設定済みのため、将来の実装候補として整理しておく
- 進め方の軸
  - 「メールアドレスを忘れた場合」の復旧導線とは切り分け、まずはログイン済みユーザー向けの設定変更として扱う
  - 変更後メールの確認フロー、再認証要否、UI 上の説明文をセットで決める
  - 問い合わせ件数や運用負荷が増えるまでは、未ログイン時のメールアドレス復旧導線は優先しない
