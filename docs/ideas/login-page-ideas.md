# Login Page Ideas

ログイン画面の現状整理と、次に整えたいポイントのメモ。

## 現状

- 未ログイン時は `src/App.tsx` から `src/features/backlog/components/LoginPage.tsx` を表示している
- 画面は 2 カラム構成で、左に「プレビュー説明」、右に「ログインフォーム」を置いている
- フォーム初期値に `akari@example.com` / `password123` を入れており、そのままローカル seed データへログインできる
- メッセージ表示は 1 本だけで、送信中・成功・失敗を同じテキスト領域で表現している
- セッション確認中は `App.tsx` 側で「読み込み中」とだけ表示し、ログイン画面とは別の見た目になっている

## 今の画面で気になる点

### 1. ローカル開発向け情報が強く、プロダクトの入口としては文脈が弱い

- 「PREVIEW」「ローカルログイン」「seed 済み backlog を確認」といった文言が前面に出ている
- 開発用としては便利だが、将来的な本番ログイン画面の情報設計とはズレやすい
- 何ができるアプリかよりも、どう入るかに説明が偏っている

### 2. 認証状態ごとの体験が分断している

- セッション確認中の画面と未ログイン画面でレイアウトが切り替わる
- 読み込み中だけ素朴なプレースホルダーになっていて、体験の連続性が弱い

### 3. フォームの状態表現が最小限

- 送信中でもボタンは通常表示のまま
- エラーと成功が同じ見た目の文章で流れるため、視認性が高くない
- 入力エラー、認証失敗、再送の導線が分かれていない

### 4. 開発用資格情報の見せ方が固定的

- サンプルアカウントをそのまま本文に載せている
- ローカル検証には便利だが、常設 UI としてはノイズになりやすい
- 将来的には「開発環境だけ出す」「コピーボタン付きの補助情報に寄せる」など整理したい

### 5. ビジュアルの重心がやや曖昧

- 左右とも同じ強さのカードで、どちらを先に読ませたいかが少しぼやける
- タイトルは強いが、プロダクト価値や安心感につながる補足が少ない

## 整える方針

### 方針1. まず「開発用ログイン画面」として意図を明文化する

- 当面はローカル Supabase 用の入口であることを認めたうえで、画面内の役割を整理する
- 開発用情報と通常のログイン UI を同列に置かず、主従を分ける

### 方針2. 認証体験を 1 つの流れに揃える

- セッション確認中、未ログイン、送信中、失敗時でレイアウトの骨格を大きく変えない
- 少なくとも `App.tsx` のローディング表示はログイン画面寄りのスケルトンや簡易版に寄せたい

### 方針3. フォームの状態を見た目で分かるようにする

- 送信中はボタン文言変更と disabled を入れる
- 成功・失敗メッセージは Alert 系の見た目に寄せる
- 可能なら入力単位の補助やエラー理由も切り分ける

### 方針4. 補助情報は折りたたむか、環境依存で出し分ける

- サンプル資格情報は常時メイン面積を使わない形にしたい
- 例:
  - 「ローカル開発用アカウントを表示」トグル
  - `import.meta.env.DEV` のときだけ表示
  - コピーボタン付きのコンパクトな補助カード

### 方針5. 認証方式の追加は公開タイミングで再評価する

- 現段階ではメール / パスワードを整える優先度が高い
- Google などのソーシャルログインは、外部ユーザーに触ってもらう段階で検討するとよい
- 導入時は Provider 設定、コールバック URL、文言整理に加えて、公開形態によっては OAuth 同意画面まわりの確認も必要になる

## たたき台の改善案

### 案1. 最小整理

- 現行レイアウトを活かしつつ文言と状態表示だけ整える
- やること
  - 「PREVIEW」「ローカルログイン」を用途が分かる文言に調整
  - 送信中 UI とエラー表示を追加
  - 読み込み中画面をログイン画面と近い見た目に揃える

向いているケース:
短時間で見た目と使い勝手のノイズを減らしたいとき

### 案2. 開発用情報をサブ要素へ退避

- 主役をフォームにして、サンプル資格情報は補助 UI に下げる
- やること
  - 左カラムはプロダクト紹介または価値訴求に寄せる
  - 開発用資格情報はアコーディオンや注釈領域へ移す
  - 本番ログイン画面へ育てやすい構成にする

向いているケース:
今後もログイン画面を育てていく前提があるとき

### 案3. 開発用と本番想定を分離

- 開発時だけ補助導線を追加し、通常 UI からは切り離す
- 例:
  - 開発環境のみデモログインボタンを出す
  - サンプル資格情報は docs 側に逃がす

向いているケース:
実運用向けの体験を早めに固めたいとき

## おすすめの進め方

まずは案1でノイズを減らし、その後必要なら案2へ進むのがよさそう。

1. 送信中 / 成功 / 失敗の状態表現を追加する
2. セッション確認中の表示をログイン画面に寄せる
3. 文言から「ローカル検証用」と「アプリの価値訴求」を整理する
4. サンプル資格情報の出し方を縮小する

## 実装時の確認ポイント

- shadcn/ui の `Alert` や既存トークンで収まるか
- モバイル幅で 2 カラム前提の情報量が重すぎないか
- 認証失敗時に Supabase の生メッセージをそのまま出すか、文言を丸めるか
- 開発用資格情報を UI に残すなら、本番ビルドでどう扱うか
- ソーシャルログインを足す場合、現段階で本当に必要か、公開範囲に対して運用コストが見合うか
