# Feature Ideas

今後の機能アイデアや設計メモを集約する場所。

## レコメンドキャッシュ見直し

### 背景

- レコメンドモーダルは `similar` と `trending` を元に候補を作っている
- 既存 backlog に入っている作品を除外するため、表示された作品を追加していくと弾切れしやすい
- TMDb API を毎回多めに叩く構成にはしたくない

### 検討した案

#### 案1: `works` 保存時に似た作品もまとめて取得して保存する

- 内容
  - `works` 登録時に TMDb の `similar` を取得し、関連テーブルや JSON として持つ
- メリット
  - レコメンド表示時の API 呼び出しを減らせる
  - 作品追加直後に関連候補が揃っている
- デメリット
  - 保存処理が重くなりやすい
  - 失敗時に作品登録とキャッシュ更新の責務が混ざる
  - レコメンドを見ない作品まで先回りで取得することになる

#### 案2: レコメンド表示時に DB キャッシュを参照し、欠損時だけ TMDb を叩く

- 内容
  - まず DB のキャッシュを読む
  - 期限切れまたは未取得の seed だけ TMDb から補充する
  - 取得結果は DB に保存して次回以降に再利用する
- メリット
  - API 使用量を抑えやすい
  - 追加処理は軽いまま保てる
  - 実際に必要になった作品だけキャッシュできる
- デメリット
  - 初回表示時だけ取得コストがかかる
  - キャッシュ期限管理が必要

#### 案3: 定期ジョブで `trending` や人気作品だけ事前に温める

- 内容
  - `trending` は定期ジョブでまとめて取得して保存する
  - `similar` はオンデマンドで取得する
- メリット
  - 空検索や初回表示が速くなりやすい
  - グローバルな候補と作品依存の候補を分けて管理できる
- デメリット
  - 定期実行の仕組みが必要
  - どこまで温めるかの運用ルールが要る

### 推奨方針

案2を基本にして、必要なら案3を足す。

- `works` 保存時にはレコメンド取得を同期でぶら下げない
- `similar` はオンデマンドで DB キャッシュする
- `trending` は DB キャッシュ化しつつ、必要なら後から定期更新に広げる
- UI では stale-while-revalidate 的に扱い、古いキャッシュを返しつつ裏で更新できる形を目指す

### たたき台のテーブル案

#### `work_recommendation_cache`

- `source_tmdb_media_type`
- `source_tmdb_id`
- `recommended_tmdb_media_type`
- `recommended_tmdb_id`
- `rank`
- `source`
  - `similar` など
- `fetched_at`
- `expires_at`

用途:
作品ごとの類似候補キャッシュ

#### `tmdb_trending_cache`

- `window`
  - `week` など
- `tmdb_media_type`
- `tmdb_id`
- `rank`
- `fetched_at`
- `expires_at`

用途:
トレンド候補のグローバルキャッシュ

### 実装イメージ

1. レコメンド表示時に seed 作品を決める
2. DB キャッシュから候補を取得する
3. 不足分や期限切れ分だけ TMDb を叩く
4. 取得結果をキャッシュテーブルに保存する
5. `trending` と `similar` をマージして既存 backlog を除外する

### 期限の目安

- `trending`: 12〜24時間
- `similar`: 7日程度

### メモ

- `works` に JSON で抱え込むより、別テーブルのほうが更新・再利用・分析がしやすい
- まずは DB キャッシュ導入だけで十分効果がありそう
- 将来的に `discover` 系やジャンルベース推薦を混ぜる場合も、この構成のほうが拡張しやすい
