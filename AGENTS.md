# AGENTS

## 基本原則

- 依存追加・削除や各種実行は `pnpm` や `npm` ではなく `vp` を使う。
- 変更前後で必要なテストの有無を確認し、未整備なら過剰でない範囲で追加する。
- `vp check --fix` は pre-commit で実行されるため、作業完了時に手動では実行しない。
- 変更後はコミットまで行い、pre-commit の通過を確認する。
- pre-commit で自動修正できないエラーが出た場合は、その内容を修正して再コミットする。
- `vp check` / `vp check --fix` を手動実行するのは、ユーザーが明示的に要求した場合、またはコミット失敗時の原因切り分けが必要な場合に限る。
- 依存追加・削除、リファクタリング、ファイル削除の変更では `vp run knip` の実行を検討し、未使用 export / file / dependency が増えていないか確認する。
- テストがある変更では、関連する `vp test` を実行する。範囲が判断しづらい場合は `vp test` を実行する。
- `main` には直接コミットしない。
- 変更は作業ブランチ上でコミットし、PR 経由で `main` に取り込む。
- 新しい作業は、最新の `main` から作成した専用 worktree 上で始める。
- 既存作業の継続では、その作業用 worktree とブランチを使う。別作業になった場合のみ新しい worktree と作業ブランチを作成する。
- ブランチ名は作業開始時点の内容に基づいて付け、作業途中で内容が多少変わっても原則として変更しない。別作業になった場合のみ新しい作業ブランチを作成する。
- 変更後は適切な単位でコミットする。

## 作業開始手順

通常の開始フローでは `.agents/skills/worktree-start/SKILL.md` を参照し、この節はその正本ルールとして扱う。

1. 現在の worktree とブランチを確認し、それが今回の作業を継続するための専用 worktree / 作業ブランチかを判定する。
2. 既存作業の継続であり、その worktree とブランチが適切な場合は、新しい worktree やブランチを作らず、現在の作業用 worktree 上で作業を続ける。
3. 新規作業を始める場合、または現在の worktree / ブランチが不適切な場合
   （例: `main`、無関係なブランチ、すでにマージ済みのローカルブランチ、他作業と共有している worktree）は、
   元のリポジトリの `main` に移動する。
4. `main` で `git fetch origin` を実行し、`origin/main` に差分があれば
   `git pull --ff-only origin main` で fast-forward のみで最新化する。
5. 調査・実装を問わず、最新の `main` を基準に `AGENTS.md` と作業対象を確認する。
6. 新規作業では、ブランチ命名規則に従って最新の `main` から作業ブランチを作成し、そのブランチ用の worktree を作成してから作業を始める。
7. 新しい worktree を作成したら、その worktree 上で依存が未展開なら `vp install` を実行する。

既存作業を継続する場合でも、必要に応じて最新の `main` を参照し、調査結果や影響範囲の確認が古い前提に依存しないようにする。

コーディングエージェントを含むすべての作業者は、この手順に従い、`main` のまま編集やコミットをせず、新規作業を既存 worktree に混在させない。

作業を完全に畳みたいときだけ `.agents/skills/worktree-cleanup/SKILL.md` を使う。

## ブランチ命名

1. ブランチ名は `feat/...` `fix/...` `chore/...` `docs/...` `refactor/...` `test/...` `ci/...` などの種別を先頭に付ける。
2. Issue がある作業では `type/<issue-number>-<slug>`、Issue がない作業では `type/<slug>` の形式を使う。

## 実装後チェック

通常の実装完了時は `.agents/skills/self-review/SKILL.md` を参照し、この節はその正本ルールとして扱う。

1. `main` または作業元ブランチとの差分を確認し、不要な変更が混ざっていないかを確認する。
2. 変更内容をセルフレビューし、挙動の破壊・運用上の回帰・説明不足がないかを確認する。
3. 変更に応じて必要な検証を実行し、実行したコマンドと結果を整理する。
4. 未実施の検証や確認できていないリスクがあれば、明示したうえでコミットする。
5. セルフレビューで重大な懸念が見つかった場合は、修正してからコミットまたは追加コミットする。

コーディングエージェントを含むすべての作業者は、完了報告の前にこの手順を実施し、懸念の有無を明示する。

## コミット

1. コミットメッセージは Conventional Commits の接頭辞を付け、日本語の体言止めにする。
2. コミットメッセージの例: `feat: シーズン追加対応の実装`, `fix: シーズンタイトル補正の修正`

## コマンド

```bash
vp dev          # 開発サーバー起動
vp build        # 本番ビルド（型チェック込み）
vp preview      # 本番ビルドのプレビュー
vp check        # Lint（型チェックあり）
vp check --fix  # Lint + 自動修正
vp run knip     # 未使用 export / file / dependency の検知
vp test         # テスト全件実行
vp test src/features/backlog/data.test.ts  # 特定ファイルのテスト実行
```

`vp` は vite-plus (`@voidzero-dev/vite-plus-core`) が提供するカスタム CLI。`pnpm` / `npm` の代わりに使う。
